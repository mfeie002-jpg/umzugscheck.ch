### 3.1 Technical SEO Audit

#### 3.1.1 Title Tag Architektur

**Der Title Tag ist der stärkste On-Page-Rankingfaktor.**

| Kriterium | Status | Details | Empfehlung |
|-----------|--------|---------|------------|
| **Ist-Zustand** | ❌ | "Umzugsfirmen vergleichen Schweiz \| Bis zu 40% sparen" | Dynamisch injizieren |
| **Diagnose** | KRITISCH | Homepage-Titel, nicht Zürich-Titel | Title Tag ist Duplikat |
| **Keyword Position** | ❌ | "Zürich" fehlt komplett | "Zürich" = 1. Wort |
| **Länge** | ⚠️ | ~70 Zeichen (zu lang für Mobile) | Max. 60 Zeichen (580px) |
| **Einzigartigkeit** | ❌ | Duplikat der Homepage | Dynamische Injection |

**Impact**: Ohne "Zürich" im Title Tag ist Top-3-Ranking mathematisch unwahrscheinlich.


### 3.2 Schema.org & Strukturierte Daten

#### 3.2.1 Das LocalBusiness Dilemma

**Problem**: Wenn die Kategorieseite "Zürich" das Schema der Umzugscheck-Zentrale trägt, sagt man Google: "Diese Seite repräsentiert das physische Büro von Umzugscheck in Zürich".

**Technisch korrekt**, aber **semantisch suboptimal** für eine Liste von Fremdfirmen.

**Solution**: Implementierung von **ItemList Schema**.

```json
{
  "@context": "https://schema.org/",
  "@type": "ItemList",
  "itemListElement": [
    {
      "@type": "ListItem",
      "position": 1,
      "name": "Swiss Move Zürich",
      "url": "https://umzugscheck.ch/firma/swiss-move-zuerich"
    },
    {
      "@type": "ListItem",
      "position": 2,
      "name": "Premium Move Zürich",
      "url": "https://umzugscheck.ch/firma/premium-move-zuerich"
    }
  ]
}
```

**Nutzen**: Erhöht die Chance auf "Carousel"-Snippet in mobilen Suchergebnissen um ~40%.

---

#### 3.2.2 FAQPage Schema

**Status**: Seite enthält einen Abschnitt "Häufig gestellte Fragen".

**Potenzial**: Fragen wie "Was kostet ein Umzug in Zürich?" sind prädestiniert für:
- Voice Search
- Featured Snippets
- "People Also Ask"-Boxen

**Anforderung**: Quellcode muss validiert werden, dass Fragen in FAQPage JSON-LD gewrappt sind.


## 7. GLOBALE TECHNISCHE ANALYSE & CORE WEB VITALS

### 7.1 Das Client-Side Rendering (CSR) Risiko

**Annahme**: Umzugscheck.ch ist eine SPA (React/Vue/Angular).

| Phase | Konsequenz |
|-------|-----------|
| **Bot empfängt** | Fast leeres HTML-Gerüst |
| **JS-Rendering** | Dauert länger, kostet Crawl Budget |
| **Rendering-Fehler** | Google sieht leere Seite → nicht indexiert |

**Evidenz**: Wiederholte "Website inaccessible" Meldungen bei Deep Links.

**Lösung**: **Server-Side Rendering (SSR)** oder **Dynamic Rendering**.

---

### 7.2 Mobile SEO & Viewport

**Risiken**:
- Tabellen sprengen Mobile-Viewport
- Buttons < 44x44 CSS Pixel
- Ungenügender Padding zwischen Touch Targets

**Checkpoints**:
```css
/* Tabellen */
table {
  overflow-x: auto;
  display: block;
}

/* Buttons */
button {
  min-width: 44px;
  min-height: 44px;
  padding: 12px 16px; /* mindestens */
}
```
